# VoiceToTextApp 项目笔记

**日期：** 2026年3月11日

## 项目简介

这是一个将语音转为文字的程序，并支持唤醒功能。项目本身没有技术门槛，主要使用现成的工具和库。

**主要缺点：** 使用的唤醒词引擎 Porcupine 是一个商业工具，存在一个月必须联网校验一次的限制。曾尝试用 Sherpa-ONNX 替代，但没有成功。

## 项目的特殊意义

此项目的所有代码均由 OpenCode 自带的 MiniMax M2.5 Free 模型生成，本人未亲手编写一行代码。这作为我个人使用 AI 生成项目的里程碑，具有一定的纪念意义。

## 经验总结

在通过 AI 辅助完成此项目的过程中，总结出以下几点经验：

1.  **先规划，后编码：** 务必先使用 `Plan` 模式，与 AI 共同总结出具体的实现步骤、技术栈等。确认无误后，再切换到 `Build` 模式，并将规划内容固化到 `Plan.md` 文件中，之后再正式开始编程。这能有效避免开发过程中偏离方向。

2.  **成本控制意识：** 使用 AI 编程确实比较耗费 tokens。例如，一次提交代码的操作，因为需要让 AI 先查看本次修改了哪些文件，会导致输入 token 较多，整个过程下来成本大约在 4 毛左右。因此，需要时刻关注 token 的消耗。

3.  **善用版本控制：** 在项目开始时，第一时间执行 `git init` 初始化 Git 仓库。每完成一个里程碑，就主动向 AI 提示提交代码。这样，如果后续出现问题，可以很方便地通过 `git reset` 来回退代码。注意，在类似 OpenCode 的环境中，可能需要主动告知 AI 去执行提交操作。

4.  **借助外部工具节省 Token：** 为了在关键的决策环节节省 Token 消耗，可以将技术方案的权衡、技术难点的预研（特别是排查复杂问题时）放在其他平台（如 Kimi 网页版）进行。在人类做好充分调研和判断后，再带着明确的方案提示给 AI，效率更高。

5.  **避免 AI 处理复杂环境问题：** 最重要的一点！尽量避免让 AI 去搭建复杂的环境或解决棘手的技术问题。AI 很可能会陷入解决问题的死循环，不断尝试各种错误的方法，从而浪费大量 Token。在这种情况下，人类的经验往往更有效。可以先由人工识别出问题的可能原因，再针对性地提示 AI。或者，将关键代码片段上传到免费的 AI 模型（如网页版大模型）进行分析，然后手动将解决方案粘贴到代码中进行测试。
